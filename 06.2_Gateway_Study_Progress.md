## 1. How to Use This Guide

Work through the phases in order. Do **not** skip Phase 0 — even if you think you know the basics, the six concepts in 06.1 Section 1 are exactly what the code is built on. Skipping them is the fastest way to get confused.

Each phase ends with a **milestone check** — a short list of things you should be able to do or explain. If you cannot answer those questions, re-read that phase before moving on.

**Time estimate per phase** (rough, varies by background):

| Phase | Background needed | Approximate time |
|---|---|---|
| Phase 0 | None | 1–3 days |
| Phase 1 | Phase 0 complete | Half a day |
| Phase 2 | Phase 1 complete | 1–2 days |
| Phase 3 | Phase 2 complete | 1–3 days depending on task |
| Phase 4 | Phase 3 complete | Ongoing |

## 3. Phase 0 — Foundations (Before You Open the Code)

These are the six ideas from 06.1 Section 1. You must understand all of them before the architecture will make sense. Each one maps directly to something in the code.

---

### 3.1 TypeScript + Node.js

**What to learn:**
- TypeScript: types, interfaces, `async`/`await`, `Promise`, generics
- Node.js: how the event loop works — why it can handle many clients without threads
- What a "long-lived server process" means (vs a script that runs and exits)

**Where to learn it:**
- TypeScript basics: [TypeScript Handbook](https://www.typescriptlang.org/docs/handbook/intro.html) — read the first 5 sections (The Basics → Everyday Types → Narrowing → Functions → Object Types)
- Node.js event loop: search "What the heck is the event loop anyway" (Philip Roberts talk) — 26 minutes, explains it better than any written doc
- async/await: MDN — "Using Promises" then "async function"

**Why it matters in the Gateway:**
The Gateway is a single Node.js process that handles hundreds of concurrent WebSocket connections without threads. If you do not understand the event loop, the non-blocking async patterns in the code will look like magic.

**Milestone check:**
- You can explain why `await` does not block other requests
- You can read a TypeScript interface and understand what fields it has
- You understand what `Promise<void>` means

---

### 3.2 HTTP vs WebSocket

**What to learn:**
- HTTP: request/response cycle, status codes, headers
- WebSocket: how it starts as an HTTP `Upgrade` request, then becomes persistent
- The key difference: HTTP is "ask and answer", WebSocket is "both sides can talk any time"

**Where to learn it:**
- MDN — "HTTP overview" (10 min read)
- MDN — "The WebSocket API" (15 min read)
- Key detail to understand: a WebSocket connection starts as a regular HTTP request with an `Upgrade: websocket` header — after that, HTTP is gone and a persistent two-way channel exists

**Why it matters in the Gateway:**
The Gateway runs both on the same port (18789). HTTP handles health checks and the control UI. WebSocket handles everything the CLI and agents do. You need to know which is which when reading the code.

**Milestone check:**
- You can explain why CLI commands use WebSocket and not REST
- You can explain what happens when a client sends `Upgrade: websocket`

---

### 3.3 JSON-RPC

**What to learn:**
- The JSON-RPC 2.0 format: `{ "id": ..., "method": ..., "params": ... }`
- How the `id` field ties a request to its response, even when many are in-flight
- What a "method" is in RPC terms (it is just a function name)

**Where to learn it:**
- [JSON-RPC 2.0 Specification](https://www.jsonrpc.org/specification) — read sections 1–5 (short, takes 15 minutes)
- Look at the examples in 06.1 Section 1.3 — these are real messages the CLI sends

**Why it matters in the Gateway:**
Every command the CLI sends (`chat.send`, `agent.list`, `config.apply`) is a JSON-RPC call. The RPC Dispatcher (the most important component) is just a lookup table of method names to handler functions.

**Milestone check:**
- You can write a valid JSON-RPC request for a method called `chat.send`
- You can explain what the `id` field is for

---

### 3.4 The `ws` Library

**What to learn:**
- How to create a WebSocket server with the `ws` npm package
- How to send and receive messages
- What `ping`/`pong` frames are (how the server detects a dead connection)

**Where to learn it:**
- [ws library README](https://github.com/websockets/ws#readme) — read the "Usage" section and the "Server example"
- You do not need to understand every detail — just enough to read code that uses `wss.on('connection', ...)` and `socket.send(...)`

**Milestone check:**
- You can read the `ws` usage in 06.1 Section 1.4 without confusion

---

### 3.5 AbortController

**What to learn:**
- What `AbortController` and `AbortSignal` are
- How to pass a `signal` into a long-running operation so it stops when aborted
- Why this pattern exists (propagating cancellation deep into async code)

**Where to learn it:**
- MDN — "AbortController" (10 min read)
- Look at the example in 06.1 Section 1.5 — it is exactly how the Channel Manager uses it

**Why it matters in the Gateway:**
Every channel (Slack, Teams, etc.) runs an infinite retry loop. The `AbortController` is the only way to stop that loop cleanly. When a channel is removed from config, `ac.abort()` is called and the loop exits.

**Milestone check:**
- You can explain how calling `ac.abort()` stops a `sleepWithAbort()` call inside a channel loop
- You understand why a plain `while(true)` loop without a signal would be hard to stop

---

### 3.6 Exponential Backoff

**What to learn:**
- Why you should not retry immediately on failure (thundering herd / server load)
- The backoff formula: `wait = initialMs * 2^attempt`
- What "jitter" is and why it helps (spreads retries across time)

**Where to learn it:**
- Read 06.1 Section 1.6 — it has a clear example table
- Optional deeper read: search "AWS exponential backoff and jitter" blog post

**Milestone check:**
- You can calculate the wait time for attempt 3 using the formula in 06.1
- You can explain what happens at attempt 11 (MAX_RESTART_ATTEMPTS = 10 → loop stops)

## 5. Phase 2 — Read the Source Code

Now open the actual source files. Read them in this order — it follows the request path from outside to inside.

All files are under `src/gateway/`.

---

### Reading order and focus points

**Step 1 — `server-constants.ts`**

Read this first. It defines all the magic numbers (timeouts, buffer sizes, limits). When you see `MAX_PAYLOAD_BYTES` in another file, you will know what it means.

Focus: understand what each constant controls. No logic here — just values.

---

**Step 2 — `server.impl.ts` (entry point)**

The orchestrator. Do not try to understand every line — just identify:
- The 9 startup steps (they match 06.1 Section 5.1)
- Where each component is created
- Where the HTTP and WebSocket servers are started
- The shutdown handler (what happens when the process gets a SIGTERM signal)

---

**Step 3 — `server-http.ts`**

Short file. Identify:
- The 5 HTTP routes (health, UI, hooks, OpenAI proxy, canvas)
- Where the WebSocket upgrade is handed off

---

**Step 4 — `server-ws-runtime.ts` → `server/ws-connection.ts`**

Read `server-ws-runtime.ts` first — it sets up the WebSocket server.
Then read `ws-connection.ts` — it handles one client's lifecycle.

Focus on the `GatewayWsClient` interface. Every RPC handler receives this object. Knowing its fields tells you what every handler knows about "who is calling".

---

**Step 5 — `auth.ts` + `auth-rate-limit.ts`**

Read `auth.ts` for the 4 auth modes.
Read `auth-rate-limit.ts` for the sliding-window limiter (matches the pseudo-code in 06.1 Section 7.3).

Compare the code to the pseudo-code — they should match. If they don't, you have found either an inconsistency in the docs or something you misunderstood.

---

**Step 6 — `server-methods.ts`** ← Most important file

This is the routing table. Spend the most time here.

- Look at how handler groups are merged into `coreGatewayHandlers`
- Find `authorizeGatewayMethod()` — trace how a `chat.send` call passes or fails
- Find `NODE_ROLE_METHODS` — understand what a "node" client can and cannot do
- Find `METHOD_SCOPE_GROUPS` — the full map of which scope is required for each method

---

**Step 7 — `server-broadcast.ts`**

Short file. Identify:
- How `seq` is incremented
- What `dropIfSlow` does
- How scope gating works for `exec.approval`

---

**Step 8 — `server-channels.ts` + `channel-health-monitor.ts`**

Read `server-channels.ts` for the three Maps (`aborts`, `tasks`, `runtimes`) and `runChannelLoop()`.

Read the backoff formula in the code and compare it to the formula in 06.1 Section 7.2. They should match.

Then read `channel-health-monitor.ts` for the zombie detection logic.

---

**Step 9 — `config-reload.ts`**

Find `diffConfigPaths()` and the lists of hot-reloadable vs restart-required paths. Compare to 06.1 Section 7.4.

---

**Step 10 — `server-lanes.ts` + `server-startup.ts`**

`server-lanes.ts`: find the three lane names and where semaphore limits are applied.
`server-startup.ts`: identify the three sidecar services.

---

### Phase 2 Milestone Check

1. Open `server-methods.ts`. Find a method handler for `chat.send`. What arguments does it receive?
2. Open `ws-connection.ts`. Find where `GatewayWsClient` is populated. What is `isLoopback` used for?

## 6. Phase 3 — Edit the Code

This phase is task-based. Pick the task that matches what you need to do, follow the steps.

---

### Task A — Add a new RPC method

**Example:** Add a method `ping.echo` that returns whatever params were sent.

1. Open `server-methods.ts`
2. Create a new handler function:
   ```typescript
   async function handlePingEcho(params: unknown, ctx: GatewayRpcContext) {
     return { echo: params };
   }
   ```
3. Add it to the handler registry:
   ```typescript
   const coreGatewayHandlers = {
     ...chatHandlers,
     'ping.echo': handlePingEcho,   // ← add here
     // ...
   };
   ```
4. Add the scope entry in `METHOD_SCOPE_GROUPS` if it requires auth:
   ```typescript
   'ping.echo': 'operator.read',
   ```
5. Rebuild and test by connecting a WebSocket client and calling `{ "id": 1, "method": "ping.echo", "params": { "hello": "world" } }`

**Key files:** `server-methods.ts` only.

---

### Task B — Change a timing threshold

**Example:** Change the handshake timeout from 3 seconds to 10 seconds.

1. Open `server-constants.ts`
2. Find `DEFAULT_HANDSHAKE_TIMEOUT_MS`
3. Change the value: `10_000`
4. Rebuild — no other files need to change

All thresholds are in `server-constants.ts`. Never hardcode a number in another file.

---

### Task C — Change the channel backoff policy

**Example:** Reduce the initial wait from 5 seconds to 2 seconds.

1. Open `server-channels.ts`
2. Find `CHANNEL_RESTART_POLICY`
3. Change `initialMs: 5_000` to `initialMs: 2_000`
4. Rebuild and test by simulating a channel disconnect

---

### Task D — Add a new hot-reloadable config path

**Example:** Make changes to `notifications.*` apply without a restart.

1. Open `config-reload.ts`
2. Find the array or set of hot-reloadable path prefixes
3. Add `'notifications'` to that list
4. Test by changing a `notifications` field in your config while the Gateway is running — it should apply without restart

---

### Task E — Add a new sidecar service

**Example:** Start a background job when the Gateway boots.

1. Open `server-startup.ts`
2. Find where the existing sidecars (browser controller, Gmail watcher, session lock GC) are started
3. Add your new service in the same pattern — start it, store any cleanup handle, and call cleanup in the shutdown hook

---

### Before editing — development workflow

```bash
# 1. Install dependencies (once)
npm install

# 2. Run in development mode (auto-rebuilds on save)
npm run dev

# 3. Run tests
npm test

# 4. Type-check without building
npx tsc --noEmit
```

Always run `npx tsc --noEmit` after a change before testing. TypeScript will catch type errors immediately — much faster than runtime debugging.

## 7. Phase 4 — Maintain the Gateway

This phase covers day-to-day operations: monitoring, diagnosing problems, applying config changes safely.

---

### Checking if the Gateway is healthy

```bash
# Basic health check — should return { "status": "ok" }
curl http://localhost:18789/health

# Full status via CLI
openclaw gateway status

# Restart the gateway if it has crashed
openclaw gateway restart
```

If `/health` returns anything other than 200, the HTTP server itself is broken — check the process logs.

---

### Reading logs

The Gateway logs structured JSON to stdout. Key fields to look for:

| Log field | What it means |
|---|---|
| `level: "error"` | Something failed — always investigate |
| `channel: "slack"` | Log line is about the Slack channel |
| `attempt: 3` | This channel has failed and retried 3 times |
| `method: "chat.send"` | An RPC call is being handled |
| `connId: "abc..."` | Which client connection this is |
| `zombie: true` | Health monitor detected a stale channel |

---

### Handling a zombie channel

A zombie channel is one that reports `connected` but has produced no events for 30 minutes. The health monitor restarts it automatically, but you may see it in logs before the restart fires.

**Manual restart of a single channel:**

```bash
openclaw gateway channels restart slack
```

**If a channel exceeds 10 restarts per hour**, it stops retrying. You will see `maxRestarts exceeded` in the logs. Fix the underlying cause (bad credentials, network issue) before restarting manually.

---

### Applying config changes safely

| Change type | How to apply | Downtime? |
|---|---|---|
| Add/remove a channel | Edit `openclaw.json` — hot reload applies it | No |
| Change a model alias | Edit `openclaw.json` — hot reload applies it | No |
| Change token scopes | Edit `openclaw.json` — hot reload applies it | No |
| Change the auth token | Edit `openclaw.json` — triggers full restart | Brief (seconds) |
| Change the port | Edit `openclaw.json` — triggers full restart | Brief |
| Change TLS settings | Edit `openclaw.json` — triggers full restart | Brief |

Config hot reload uses `chokidar` to watch the file. Changes are debounced by 300ms — save the file once and wait.

---

### Adding a new channel at runtime (no restart)

1. Edit `openclaw.json` — add the new channel config under `channels:`
2. Save the file
3. The Config Reloader detects the change within 300ms
4. The Channel Manager starts the new channel loop automatically
5. Confirm with: `openclaw gateway channels status`

---

### Adjusting rate limits

Rate limits are in `auth-rate-limit.ts`. If you need to change them:
1. Edit the constants (`WINDOW_MS`, `MAX_ATTEMPTS` for each scope)
2. Rebuild and restart (rate limiter config is not hot-reloadable — it is structural)

Loopback addresses (localhost) are always exempt from rate limiting.

---

### What to do when the Gateway refuses to start

Work through these checks in order:

| Symptom | Check |
|---|---|
| `EADDRINUSE` error | Port 18789 is already in use — find and stop the other process |
| `invalid token` at startup | Auth token in config is malformed — check `auth.token` |
| Channel immediately fails | Check credentials for that channel (bot token, signing secret, etc.) |
| Process starts but CLI cannot connect | Firewall blocking port 18789, or wrong `bindHost` in config |
| Config reloader crashes | Config file has invalid JSON — validate with `jq . openclaw.json` |

---

### Phase 4 Milestone Check

1. You can check whether the Gateway is running and healthy using a single curl command
2. You can identify a zombie channel in the logs and restart it manually
3. You know which config changes require a restart vs which are hot-reloaded
4. You can add a new channel to a running Gateway without stopping it
